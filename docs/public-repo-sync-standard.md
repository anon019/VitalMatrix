# 公开远程仓库同步标准

适用范围：
- 本地工作目录同步到公开远程仓库
- 当前重点适用于 `VitalMatrix` 这类公开 GitHub 仓库

## 1. 目标

公开仓库只保留可公开的代码、模板、文档和通用部署示例。
任何与本地机器、真实线上环境、私有资产绑定的内容，都必须在同步前清理、模板化或排除。

## 2. 绝对禁止进入公开仓库的内容

以下内容禁止直接 push 到公开仓库：

- `.env`、API Key、OAuth Token、数据库密码、JWT Secret
- `uploads/`、本地照片、用户数据、数据库备份
- 私有证书、密钥文件、`*.pem`、`*.key`、`*.p12`
- 本地调试脚本、一次性排障脚本、硬编码用户 ID 的脚本
- `project.private.config.json`
- 带真实 AppID 的 `project.config.json`
- 本机绝对路径，例如 `/root/health/...`
- 真实线上域名、真实服务器目录、真实内部 Referer

## 3. 必须模板化或参数化的内容

以下内容如果需要保留在公开仓库，必须改成模板或环境变量驱动：

- 域名：统一使用 `https://your-domain.example.com` 一类占位值
- OAuth 回调地址：必须使用占位域名或环境变量
- CORS 白名单：只能保留通用示例，不能写真实线上域名
- OpenRouter / 第三方请求头中的站点标识：必须可配置
- WeChat AppID：只保留 `project.config.json.example`
- 部署脚本中的目录、日志路径、虚拟环境路径：必须使用相对路径或环境变量
- Nginx / systemd / shell 脚本：必须是公开可复用模板，不能绑定单台机器目录

## 4. 可以保留的硬编码

以下内容可以保留，因为它们属于公开、稳定、非私有的上游地址：

- 官方第三方 API 地址
  - Polar
  - Oura
  - WeChat 官方接口
  - OpenRouter / DeepSeek / OpenAI 等官方服务地址
- 本地开发默认地址
  - 例如 `http://localhost:8000`
  - 前提是它明确属于本地开发或本机健康检查用途

## 5. 小程序公开仓库标准

- 不提交 `project.config.json`
- 保留 `project.config.json.example`
- `utils/config.js` 只能写占位域名或本地可修改配置
- 如果后端启用了 `WEB_ACCESS_PASSWORD`，小程序侧只能保留空的密码位，不能提交真实密码
- 所有接口路径必须和当前后端真实路由保持一致

## 6. 后端公开仓库标准

- `config.py` 中不得保留真实线上域名
- `.env.example` 只能出现占位值，不得出现真实配置
- shell 脚本必须优先使用：
  - 脚本相对路径
  - 环境变量
  - 模板目录
- 不在公开仓库里保留只能在单台服务器上运行的硬编码路径

## 7. 文档与部署文件标准

- README、docs、deploy 配置中的域名必须是模板值
- 所有示例命令都必须是公开可执行或可替换的通用示例
- 如某部署文件强依赖实际环境，必须明确标注“模板，需要替换域名/路径/证书路径”

## 8. 公开仓库同步流程

后续同步公开仓库时，固定按以下流程执行：

1. 不直接改本地工作仓库的 `origin`
2. 单独克隆公开远程仓库到临时目录
3. 新建同步分支，不直接在远程 `main` 上操作
4. 只同步经过批准的目录和文件
5. 同步前做排敏和模板化处理
6. 提交身份固定使用 GitHub 用户名 + no-reply 邮箱
7. push 到同步分支后，再 fast-forward 合并到 `main`
8. 合并完成后删除远程同步分支

## 9. 每次 push 前的检查清单

每次同步前，至少检查以下项目：

- 是否包含 `.env`
- 是否包含 `uploads/`、照片、本地资产
- 是否包含真实域名
- 是否包含本机绝对路径
- 是否包含带真实值的 `project.config.json`
- 是否包含私有脚本或临时调试文件
- 是否有接口路径已经和当前后端路由漂移
- 是否有部署脚本仍然依赖固定目录

## 10. 处理原则

- 能模板化的，改成模板后再公开
- 不能安全公开的，直接排除，不上传
- 不为“保持完全一致”而上传本地私有资产
- 公开仓库以“可公开、可复用、可维护”为第一目标，不以“和本地环境一字不差”为目标

## 11. 后续执行约束

从现在开始，后续所有远程公开仓库 push 默认遵循本标准。
如果某次同步需要例外处理，必须先明确说明例外项，再决定是否同步。
